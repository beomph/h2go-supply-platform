프로젝트명: AI 기반 H2GO 수소 공급망 플랫폼 고도화
서비스명(브랜드): H2GO 수소 거래 플랫폼 — AI 엔진
부제: 머신러닝/딥러닝을 활용한 수요 예측, 배차 최적화, 이상 탐지, 지능형 의사결정 지원 시스템

================================================================================
1. 프로젝트 개요 및 목적
================================================================================

1.1 배경
- H2GO 플랫폼은 현재 정적 HTML/CSS/JS 기반 대시보드와 규칙 기반(rule-based)
  로직으로 수요/공급 현황을 시각화하고 있으나,
  실제 수요 예측·배차 최적화·리스크 관리는 사람의 경험과 수작업에 의존하고 있음.
- 수소 공급망의 특수성(대량 저장 불가, T/T 운송, 잔압 기반 정산단가 변동,
  MTOP/MSFO 계약 조건)을 고려할 때, 데이터 기반의 AI 의사결정 엔진이
  비용 절감과 운영 효율화의 핵심 레버가 됨.

1.2 목적
- 기존 H2GO 플랫폼에 **머신러닝(ML) 및 딥러닝(DL) 파이프라인**을 통합하여
  다음 4가지 AI 핵심 역량을 구현한다:
  1) 수요 예측 AI: 충전소별 시간대/일별/주별 수소 수요를 정밀 예측
  2) 배차 최적화 AI: T/T·기사 자원 배분 및 경로를 자동 최적화
  3) 이상 탐지 AI: 설비 고장, 공급 중단, 비정상 소비 패턴을 조기 감지
  4) 지능형 챗봇/어시스턴트: LLM 기반 도메인 특화 질의응답 및 운영 가이드

1.3 현행 웹 로그인(비-AI 범위, 참고)
- 로그인 시 `check_login_id_registered` RPC로 `member_profiles.login_id` 가입 여부를 먼저 확인한다.
- 미가입 아이디면 Supabase Auth를 호출하지 않고 “가입되지 않은 아이디” 안내 후 종료한다.
- 가입된 아이디만 `signInWithPassword`로 비밀번호를 검증하며, 실패 시 비밀번호 오류만 안내한다.
- Supabase Auth: **Email 가입 허용**(Allow new users to sign up) 켜기. 꺼 있으면 `Email signups are disabled`. Confirm email은 끄고, `scripts/supabase_mailer_autoconfirm.ps1` 로 `disable_signup`+`mailer_autoconfirm` 동시 설정 가능.
- `member_profiles`: `business_parties[]`(공급자·운송자·수요자 복수), `username`, `authority`, `login_id`. 공급자→판매 대시보드, 수요자→구매, 둘 다면 전환 가능. 운송자만→구매만.

================================================================================
2. 핵심 문제 정의 (AI 관점)
================================================================================

2.1 수요 예측의 부정확성
- 현재: 과거 평균 또는 운영자 경험에 의한 발주량 결정
- 문제: 요일·시간대·계절·이벤트에 따른 수요 변동을 반영하지 못해
  재고 과다(잔압 증가 → 정산단가 인상) 또는 재고 부족(판매 중단) 발생
- AI 해결: 시계열 예측 모델(LSTM, Transformer 등)로 다변량 수요 예측

2.2 배차 계획의 비효율
- 현재: 단순 규칙(선입선출, 근거리 우선) 기반 수동 배차
- 문제: 공차 운행, 대기 시간, 비효율 경로로 운송원가 증가
- AI 해결: 강화학습(RL) 또는 메타휴리스틱 기반 동적 배차 최적화

2.3 설비/공급 이상의 늦은 감지
- 현재: 고장·이상이 발생한 후에야 수동 보고로 인지
- 문제: 긴급 대응 지연 → 연쇄적 공급 차질
- AI 해결: 센서/운영 데이터 기반 이상 탐지(Anomaly Detection) 모델

2.4 운영 지식의 암묵성
- 현재: 숙련 운영자의 경험에 의존하는 의사결정
- 문제: 인력 변동 시 운영 품질 저하, 신규 인력 학습 비용 증가
- AI 해결: LLM 기반 도메인 특화 어시스턴트로 운영 지식 체계화

3. AI 모듈 상세 설계
================================================================================

3.1 수요 예측 엔진 (Demand Forecasting Engine)
────────────────────────────────────────────────
- 목표:
  - 충전소별 향후 1~7일 시간대별 수소 수요량 예측
  - 예측 정확도: MAPE 15% 이내 (MVP), 10% 이내 (고도화)

- 입력 피처(Features):
  - 과거 판매 이력: 충전소별 시간대/일별 수소 판매량 (최소 6개월)
  - 시간 피처: 요일, 시간대, 공휴일, 계절
  - 외부 변수: 기온, 강수량, 미세먼지 (수소차 이용 패턴에 영향)
  - 수소차 등록 대수 증가 추이 (월별)
  - 충전소 운영 정보: 영업시간, 휴무일, 충전기 수, 지역 특성

- 모델 아키텍처 (단계별):
  - Phase 1 (ML 베이스라인):
    - LightGBM / XGBoost 기반 테이블형 시계열 예측
    - 피처 엔지니어링: 래그(lag) 피처, 롤링 통계량, 시간 임베딩
    - 장점: 빠른 학습, 해석 가능성, 적은 데이터에서도 동작
  - Phase 2 (DL 고도화):
    - LSTM / GRU 기반 다변량 시계열 모델
    - Temporal Fusion Transformer (TFT) 적용
    - Attention 메커니즘으로 피처 중요도 해석
  - Phase 3 (앙상블/AutoML):
    - ML + DL 앙상블 (Stacking)
    - 충전소별 자동 모델 선택 (AutoML 파이프라인)

- 출력:
  - 시간대별 예측 수요량 (포인트 추정 + 신뢰구간)
  - 피크 시간대 경고 알림
  - 안전 재고 기반 자동 발주 제안 (발주량, 발주 시점)

- 학습/서빙 파이프라인:
  - 학습: 일 1회 배치 재학습 (새로운 판매 데이터 반영)
  - 서빙: REST API (Flask/FastAPI) → 대시보드 연동
  - 모델 레지스트리: MLflow로 버전 관리, A/B 테스트 지원

- 평가 지표:
  - MAPE (Mean Absolute Percentage Error)
  - RMSE (Root Mean Squared Error)
  - 피크 시간대 예측 정확도 (별도 집계)
  - 재고 부족/과잉 발생 빈도 (비즈니스 KPI와 연동)

3.2 배차 최적화 엔진 (Dispatch Optimization Engine)
────────────────────────────────────────────────────
- 목표:
  - T/T·기사 자원을 활용하여 전체 운송비 최소화 + 수요자 만족도 최대화
  - 공차 운행 비율 30% 감소 (MVP 기준)

- 입력:
  - 수요 예측 엔진의 출력 (충전소별 필요 시간대·물량)
  - T/T 차량 정보: 용량, 현재 위치, 잔압, 정비 상태
  - 운송기사 정보: 근무 가능 시간, 현재 위치, 숙련도
  - 충전소/출하센터 위치 및 이동 시간 매트릭스
  - 제약 조건: 기사 최대 근무시간, T/T 회전 시간, 통행 제한

- 모델 아키텍처 (단계별):
  - Phase 1 (규칙 기반 + 휴리스틱):
    - Greedy 알고리즘: 잔여 재고 임계값 기반 우선순위 정렬
    - 클러스터링(K-means) 기반 지역 그룹핑 후 TSP 근사해
    - Google OR-Tools 활용 VRP(Vehicle Routing Problem) 솔버
  - Phase 2 (메타휴리스틱):
    - 유전 알고리즘(GA) 또는 Simulated Annealing 기반 VRP 최적화
    - 다목적 최적화: 운송비 최소화 + 잔압 최소화 + 서비스 레벨 유지
  - Phase 3 (강화학습):
    - Deep Q-Network (DQN) 또는 PPO 기반 동적 배차
    - 상태: 현재 T/T 위치, 충전소 잔여 재고, 시간대
    - 행동: T/T별 다음 방문 충전소 선택
    - 보상: 운송비 절감 + 재고 부족 페널티 회피
    - 시뮬레이션 환경 구축 → 오프라인 학습 → 온라인 배포

- 출력:
  - T/T별 일일 배차 스케줄 (시간, 경로, 예상 충전량)
  - 배차 계획 시각화 (지도 기반 경로 표시)
  - 대안 시나리오 비교 (비용, 서비스 레벨, 잔압 등)

- 평가 지표:
  - 총 운송 거리/시간 절감률
  - 공차 운행 비율
  - 수요자별 서비스 레벨 달성률 (요청 시간 내 배송 비율)
  - T/T 가동률 (운행 시간 / 가용 시간)

3.3 이상 탐지 엔진 (Anomaly Detection Engine)
──────────────────────────────────────────────
- 목표:
  - 설비 고장, 공급 중단, 비정상 소비 패턴을 발생 전 또는 발생 직후 감지
  - 오탐률(False Positive) 5% 이내, 미탐률(False Negative) 10% 이내

- 입력 데이터:
  - T/T 센서 데이터: 탱크 압력, 온도, 유량 (실시간 또는 배치)
  - 충전소 운영 데이터: 시간당 충전량, 가동 시간, 에러 로그
  - 출하센터 생산 데이터: 생산량, 설비 가동률, 원료 투입량
  - 거래 데이터: 발주-공급 매칭, 정산 내역의 이상 패턴

- 모델 아키텍처:
  - Phase 1 (통계 기반):
    - Z-score / IQR 기반 이상치 탐지
    - 이동 평균 대비 편차 모니터링
    - 장점: 설명 가능성 높음, 빠른 구현
  - Phase 2 (ML 기반):
    - Isolation Forest: 고차원 데이터에서 이상치 탐지
    - One-Class SVM: 정상 패턴 학습 후 이탈 감지
    - 시계열 분해(STL) + 잔차 기반 이상 감지
  - Phase 3 (DL 기반):
    - Autoencoder (AE): 정상 패턴의 저차원 표현 학습
      → 복원 오차가 임계치 초과 시 이상으로 판정
    - LSTM-Autoencoder: 시계열 패턴의 이상 탐지
    - Variational Autoencoder (VAE): 확률적 이상 스코어링

- 출력:
  - 실시간 이상 감지 알림 (심각도: 주의/경고/긴급)
  - 이상 유형 분류 (설비 고장 징후, 비정상 소비, 데이터 오류 등)
  - 이상 발생 시 권장 조치 안내
  - 이상 이력 대시보드 (빈도, 유형, 영향도 추이)

- 평가 지표:
  - Precision / Recall / F1-score
  - 오탐률 (False Positive Rate)
  - 평균 탐지 소요 시간 (MTTD: Mean Time To Detect)
  - 비즈니스 영향: 조기 감지로 예방한 손실 금액 추정

3.4 지능형 AI 어시스턴트 (LLM 기반 챗봇 고도화)
────────────────────────────────────────────────
- 목표:
  - 기존 OpenAI API 기반 단순 챗봇을 도메인 특화 지능형 어시스턴트로 고도화
  - 운영자가 자연어로 질의하면 데이터 기반 인사이트를 즉시 제공

- 기능 범위:
  - 자연어 데이터 질의:
    - "오늘 강남 충전소 수요 예측 얼마야?"
    - "이번 주 벤트 발생 가능성이 가장 높은 출하센터는?"
    - "T/T-003이 내일 몇 회 운행 예정이야?"
  - 의사결정 지원:
    - "지금 추가 발주하는 게 나을까, 다음 회차에 합치는 게 나을까?"
    - "MTOP 미달 위험이 있는 계약 목록 알려줘"
  - 운영 가이드:
    - 신규 운영자를 위한 절차 안내
    - 긴급 상황 대응 프로토콜 안내

- 기술 구현:
  - RAG (Retrieval-Augmented Generation):
    - 플랫폼 운영 매뉴얼, 계약 조건, FAQ 등을 벡터 DB에 인덱싱
    - 질의 시 관련 문서를 검색하여 LLM 컨텍스트로 주입
    - 벡터 DB: ChromaDB 또는 FAISS (로컬) / Pinecone (클라우드)
  - Function Calling / Tool Use:
    - LLM이 수요 예측 API, 배차 API, 이상 탐지 API를 직접 호출
    - 실시간 데이터를 기반으로 답변 생성
  - 프롬프트 엔지니어링:
    - 수소 공급망 도메인 시스템 프롬프트
    - Few-shot 예시로 응답 품질 안정화
    - 환각(hallucination) 방지를 위한 가드레일

- 평가 지표:
  - 응답 정확도 (사실 기반 답변 비율)
  - 사용자 만족도 (CSAT)
  - 평균 응답 시간
  - Function Call 성공률

================================================================================
4. 데이터 파이프라인 아키텍처
================================================================================

4.1 데이터 수집 계층
- 실시간 스트림:
  - T/T 센서 데이터 (압력, 온도, GPS) → MQTT/Kafka
  - 충전소 POS/충전기 데이터 → REST API 웹훅
- 배치 데이터:
  - 일별 판매 실적, 정산 내역 → CSV/DB 연동
  - 기상 데이터 (기상청 API), 수소차 등록 통계 (공공 API)

4.2 데이터 저장 계층
- 원시 데이터: PostgreSQL 또는 SQLite (MVP 단계)
- 피처 스토어: 전처리된 ML 피처 저장/버전관리
  - 시간 집계 피처, 래그 피처, 롤링 통계량
- 벡터 DB: ChromaDB (RAG용 문서 임베딩 저장)

※ MVP(현재 레포 단계)에서는 DB 연결 전 단계로, 아래 파일 기반 샘플 데이터로 개발/테스트를 시작한다.
  - 스키마 정의: `.taskmaster/docs/data_schema.md`
  - 샘플 데이터: `data/sample/*` (JSON/CSV)
  - 샘플 데이터 생성기: `scripts/generate_sample_data.py` (재현 가능한 시뮬레이션 데이터 생성)

4.3 데이터 전처리 파이프라인
- 결측치 처리: 시간대별 보간(interpolation), 충전소별 패턴 기반 대체
- 이상치 처리: IQR 기반 클리핑 + 도메인 규칙(물리적 한계 초과 값 제거)
- 피처 엔지니어링:
  - 시간 피처: hour_of_day, day_of_week, is_holiday, season
  - 래그 피처: lag_1h, lag_24h, lag_7d
  - 롤링 통계: rolling_mean_24h, rolling_std_7d
  - 외부 피처: temperature, precipitation, pm25
- 데이터 검증: Great Expectations 또는 커스텀 검증 스크립트

4.4 ML 학습/서빙 파이프라인
- 실험 관리: MLflow (실험 추적, 모델 레지스트리, 아티팩트 저장)
- 학습 스케줄:
  - 수요 예측: 매일 01:00 배치 재학습
  - 이상 탐지: 주 1회 재학습 + 온라인 임계치 업데이트
  - 배차 최적화: 이벤트 트리거 (새 주문 입력 시 재계산)
- 모델 서빙: FastAPI 기반 REST API
  - /api/v1/forecast — 수요 예측 결과 조회
  - /api/v1/dispatch — 배차 최적화 결과 조회
  - /api/v1/anomaly — 이상 탐지 상태 조회
  - /api/v1/chat — AI 어시스턴트 질의
- 모니터링: 모델 드리프트 감지 (PSI, KS 통계량), 예측 성능 추이 대시보드

6. AI KPI 및 성과 지표

6.1 모델 성능 KPI
- 수요 예측 MAPE: < 15% (MVP), < 10% (고도화)

- 배차 최적화 운송비 절감률: > 15%
- AI 어시스턴트 응답 정확도: > 90%

6.2 비즈니스 영향 KPI
- 판매 중단(재고 부족) 건수: 50% 감소
- 벤트(대기방출) 발생량: 30% 감소
- 공차 운행 비율: 30% 감소
- T/T 회전율: 20% 개선
- 평균 잔압 수준: 15% 감소 → 정산단가 절감

6.3 플랫폼 운영 KPI
- API 응답 시간: < 500ms (예측/배차), < 2s (챗봇)
- 모델 재학습 주기 준수율: 99%
- 데이터 파이프라인 가동률: 99.5%

10. UX 원칙 (AI 기능)
================================================================================

- AI 결과의 투명성:
  - 예측 결과에 신뢰구간 표시
  - "왜 이 값을 예측했는가?" 설명 제공 (피처 중요도, 유사 과거 패턴)
  - 이상 탐지 시 근거 데이터 함께 표시

- 사용자 주도권 유지:
  - AI는 "제안"하고, 최종 결정은 운영자가 함
  - AI 추천을 수정/거부할 수 있는 UI 제공
  - 수정된 결정은 피드백으로 재학습에 반영

- 점진적 신뢰 구축:
  - 초기에는 AI 추천과 실제 결과를 나란히 비교하는 "관찰 모드"
  - 정확도가 검증되면 "반자동 모드" → "자동 모드"로 전환
  - 각 단계에서 운영자 확인/승인 절차 포함

- 긴급 상황 우선:
  - 이상 탐지 알림은 즉시 눈에 띄는 UI로 표시
  - AI가 긴급 대응 시나리오를 자동 생성하여 제안
  - 푸시 알림 / SMS 연동 (향후)
